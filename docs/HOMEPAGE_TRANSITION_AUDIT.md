Repository note: this project is now a local Git repository connected to `origin` at `https://github.com/karinrub/maui-lessons.git`. The connected GitHub Pages URL is `https://karinrub.github.io/maui-lessons/`. The deployment source path is `/Users/karinrubin/Developer/maui-lessons`.

Audit only. No code was changed. Scope: `src/pages/Home.tsx`, `src/components/home/OpeningScene.tsx` + `.css`, `src/components/home/StackedServicesDeck.tsx` + `.css`, `src/layout/SiteLayout.tsx`, `src/App.tsx`, `src/hooks/usePrefersReducedMotion.ts`, `src/index.css`.

## How the system is actually built

`Home.tsx` renders `<OpeningScene /><StackedServicesDeck />` as two independent siblings. There is no parent orchestrator, no shared timeline, and no shared ScrollTrigger between them. Each component builds, times, pins, and refreshes its own GSAP world with only one deliberate coupling point: `StackedServicesDeck` watches `document.querySelector('.opening-scene')` for a `data-intro-complete="true"` attribute via `MutationObserver` before it will build its own ScrollTrigger. That is the entire "handshake" between the two halves of what is supposed to be one continuous sequence.

**OpeningScene geometry.** `.opening-scene` (the ScrollTrigger `trigger`) has `min-height: 100svh` and contains a single child, `.opening-scene__pin` (the `pin` target), which has a fixed `height: 100svh`. The ScrollTrigger is `start: 'top top'`, `end: '+=36%'`, `scrub: 1`, `pin: pin`, `anticipatePin: 1`, `invalidateOnRefresh: true`. GSAP pins `.opening-scene__pin` and inserts a pin-spacer in its place sized to `pin height + scroll distance`, so `.opening-scene` ends up rendering at roughly `100vh + 36vh ≈ 136vh` of document flow, all of it driven by one scrubbed timeline: video/frame fade out, tagline fade in word-by-word, tagline fade out, then `gsap.set(media, { visibility: 'hidden' })`.

**StackedServicesDeck geometry.** `.stacked-services-deck` (trigger) also has one child, `.stacked-services-deck__pin` (pin target). Its ScrollTrigger: `start: 'top top'`, `end: '+=234%'`, `scrub: 1`, `pin`, `anticipatePin: 1`, `invalidateOnRefresh: true` — a much longer pin (three-card cycle). There is a second, separate, non-pinned ScrollTrigger on the same trigger element driving the arched heading: `start: 'top bottom'`, `end: 'top top'`, `scrub: 1`, sliding the heading in from `xPercent: -140` to `0` over exactly the one viewport-height of natural scroll that happens right before the deck's own pin engages.

**The coupling.** The deck's pinned ScrollTrigger is not created on mount. `setupDeckTimeline` is gated: if `.opening-scene`'s `data-intro-complete` attribute is not yet `"true"`, a `MutationObserver` waits for it, then schedules `setupDeckTimeline` on the next animation frame, which itself calls `ScrollTrigger.refresh()` on the frame after that. Meanwhile, `OpeningScene` itself calls `ScrollTrigger.refresh()` synchronously inside the same GSAP timeline callback that flips `introComplete` to true (`markIntroComplete()` at line 590, `ScrollTrigger.refresh()` at line 614 — same `.call()`/synchronous block).

That ordering is the crux of the bug.

## Issues

### 1. Two independently-pinned sections with no shared timeline or orchestrator
**Description:** `OpeningScene` and `StackedServicesDeck` each own a full pin/scrub/refresh lifecycle. Nothing external coordinates start/end handoff, pin-spacer sizing, or refresh timing between them. The "one continuous cinematic sequence" the product wants is being *simulated* by two unrelated ScrollTrigger instances that happen to sit adjacent in the DOM.
**Root cause:** Architectural choice to build each section as a self-contained component with its own `gsap.context`/ScrollTrigger rather than a single home-level scroll controller (or a shared parent timeline with two labeled child timelines).
**Evidence:** `OpeningScene.tsx` lines 592–615 vs. `StackedServicesDeck.tsx` lines 98–157 — separate `gsap.timeline({ scrollTrigger: {...} })` calls, separate `gsap.context`, separate `ScrollTrigger.refresh()` calls, no shared ref or parent-level trigger.
**Severity:** High — this is the structural precondition for every other issue below. Nothing here is individually "wrong" GSAP usage, but the split-ownership model is what makes the boundary between the two sections fragile.

### 2. Cross-component readiness signal implemented as attribute + MutationObserver instead of state/callback
**Description:** The only synchronization primitive between the two components is a DOM attribute (`data-intro-complete`) read via `document.querySelector` and watched with a `MutationObserver`, rather than a prop, context value, or event emitted through React's own render cycle.
**Root cause:** The two components have no shared parent state (`Home.tsx` renders them as unconnected siblings), so the author reached for a DOM-level signal to bridge them.
**Evidence:** `StackedServicesDeck.tsx` lines 140–157; `OpeningScene.tsx` sets `data-intro-complete={introComplete}` on the section (line 858) where `introComplete` is React state updated via `setIntroComplete(true)` inside `markIntroComplete` (line 232–235), itself called from inside a GSAP `.call()` (line 590).
**Severity:** Medium-High. This works, but it introduces a real timing gap: the observer only fires *after* React commits the attribute to the DOM, which itself only happens *after* the GSAP callback that set the ref (`introCompleteRef.current = true`) returns. There is no guarantee about how many animation frames separate "GSAP callback fires" from "MutationObserver callback fires."

### 3. Refresh-ordering race between the two ScrollTriggers at the exact moment they need to hand off
**Description:** At the instant the intro finishes, `OpeningScene` builds its scroll-pinned timeline and immediately calls `ScrollTrigger.refresh()` (line 614) in the same synchronous block that flips `introComplete`. But `StackedServicesDeck`'s own ScrollTrigger — the one that determines how much extra document height the page gains from the 234%-vh pin — has not been created yet at that point, because it's waiting on the MutationObserver, which is waiting on React's commit of the `data-intro-complete` attribute, which is waiting on the *next* render pass after the state update. `OpeningScene`'s `refresh()` therefore caches `end` positions against a **shorter total document height** than the one that will exist a frame or two later, once the deck's pin-spacer is finally inserted. A second `refresh()` happens later (`StackedServicesDeck.tsx` line 131–133, inside a `requestAnimationFrame`), which corrects the layout going forward — but only for whatever ScrollTrigger state exists *after* that point.
**Root cause:** `invalidateOnRefresh: true` on both triggers means every `refresh()` recomputes geometry from current DOM measurements, but the two `refresh()` calls are not sequenced relative to each other or to the actual paint/layout in between. The dependency graph is circular: `OpeningScene`'s end position depends on total document height, which depends on whether `StackedServicesDeck`'s pin-spacer already exists, which depends on `OpeningScene` having already reported "intro complete" through a DOM attribute.
**Evidence:** `OpeningScene.tsx` line 614 (`ScrollTrigger.refresh()` inside the intro `.call()`) vs. `StackedServicesDeck.tsx` lines 131–133 (a *second*, later `ScrollTrigger.refresh()` inside its own setup, itself gated by `requestAnimationFrame` after the MutationObserver fires).
**Severity:** High. This is the strongest candidate for the actual bug. If a user reaches the boundary and reverses direction inside the narrow window between these two `refresh()` calls — or if the browser's paint timing shifts that window even slightly, which will vary by device — ScrollTrigger's cached start/end for `OpeningScene` (and hence where its pin should re-engage on the way back up) is stale relative to the deck's real, final document position. The visible symptom of a ScrollTrigger acting on stale cached geometry is exactly a gap: the pinned hero doesn't re-engage exactly where the deck's pin releases, because the two were never measured against the same layout snapshot.

### 4. Forward and reverse scroll are not actually symmetric, despite both timelines being scrubbed
**Description:** Scrubbed GSAP timelines are normally direction-agnostic — `scrub: 1` plus a `ScrollTrigger` should reverse cleanly because the animation is a pure function of scroll progress. But this codebase layers one-shot, forward-only side effects on top of that reversible core: `markIntroComplete` / `data-intro-complete` is set once and never unset; `setupDeckTimeline` runs once, guarded by `if (isCancelled || prefersReducedMotion || timelineContext) return`, and is never torn down and rebuilt if the user scrolls back above the hero and the intro state is somehow re-entered; audio muting (`muteVideoFromScroll`) only fires `if (scrollState.direction > 0)`, i.e., is deliberately one-directional.
**Root cause:** The mental model driving this code is "play an intro once, then hand off to a scroll experience," not "represent scroll position as a continuous, reversible function from 0 to 1 across the whole homepage." The former is fundamentally asymmetric; the latter is what a true continuous cinematic sequence requires.
**Evidence:** `OpeningScene.tsx` lines 232–235 (`markIntroComplete`, one-way flag), lines 237–244 (`muteVideoFromScroll`, direction-gated), `StackedServicesDeck.tsx` line 99 (`if (... || timelineContext) return`, one-time setup guard).
**Severity:** Medium. Doesn't by itself explain a *blank space*, but it's evidence of the same underlying design pattern (irreversible one-shot bootstrapping layered under a reversible scrub) that produces issue 3, and it will keep producing similar asymmetric bugs elsewhere on this section (e.g., re-entering the hero after visiting the deck does not replay any intro state, because there is none to replay — the intro is not a phase of the scroll geometry, it's a separate, prior, non-repeatable animation).

### 5. `anticipatePin` and pin-spacer sizing are locked to `svh`/`vh` units that shift under mobile browser chrome
**Description:** Both pin containers use `100svh` for their pinned height, and the deck's `end` is expressed as `+=234%` (percentage of viewport height at time of calculation). `svh` is intentionally used to avoid the classic mobile URL-bar resize problem, but ScrollTrigger's own start/end caching still happens once per `refresh()` and is not automatically re-synced to `svh` changes unless another `refresh()` (or `ScrollTrigger.refresh()` on `resize`) runs at the right moment.
**Root cause:** Reliance on refresh-triggered geometry recalculation instead of continuous, resize-observer-driven recalculation, combined with the fact that (per issue 3) refreshes are already happening at fragile, hand-tuned moments rather than being centrally owned.
**Evidence:** `OpeningScene.css` line 10 (`height: 100svh`), `StackedServicesDeck.css` line 11 (`min-height: 100svh`), `StackedServicesDeck.tsx` line 109 (`end: '+=234%'`).
**Severity:** Low-Medium on its own; compounding on mobile Safari specifically, where address-bar collapse/expand during a scroll gesture is common and would independently perturb the same geometry that issue 3 already leaves fragile.

### 6. The heading's separate, unpinned ScrollTrigger is timed to fit exactly inside a gap whose size is controlled by a different component
**Description:** The heading's slide-in (`start: 'top bottom'`, `end: 'top top'` on `.stacked-services-deck`) assumes there is a full, stable viewport-height of ordinary (non-pinned) scroll distance between "deck section's top enters the viewport" and "deck section's top reaches the viewport top, at which point its own pin engages." That distance is not an intrinsic property of the deck — it is entirely a function of where `OpeningScene`'s pin releases, which (per issues 1–3) is itself dynamically computed and refreshed asynchronously.
**Root cause:** Same root as issue 1 (no shared timeline), applied specifically to the one piece of "connective tissue" animation that's supposed to bridge the two sections.
**Evidence:** `StackedServicesDeck.tsx` lines 84–96.
**Severity:** Medium. This is the animation most likely to visibly stutter or feel like "dead scroll" even when there's no literal gap in painted content, because its timing window is borrowed from geometry it doesn't own.

## How these relate

Issues 1, 2, 3, 4, and 6 are not five independent bugs — they're one architectural decision (no shared scroll controller between the two sections) expressing itself in five places. Issue 1 is the root condition. Issue 2 is the workaround the codebase reached for because of issue 1. Issue 3 is the specific race that workaround introduces. Issue 4 is why the race is asymmetric (worse in reverse) rather than symmetric. Issue 6 is a second, independent piece of animation that inherited the same fragility as issue 3 because it also depends on a geometry boundary it doesn't control. Issue 5 is an orthogonal aggravating factor (viewport-unit volatility) that makes the existing race window less predictable, especially on mobile, but would exist even if issues 1–4 were fixed.

## Root Cause Analysis

**1. What is the single biggest architectural weakness in the current homepage transition system?**
There is no single owner of "homepage scroll." Two components each build, pin, and refresh their own ScrollTrigger and treat the other as an opaque DOM node to be queried or waited on. The handoff between them is implemented with a `MutationObserver` on a data attribute rather than a shared timeline, shared refs, or a parent controller — so the exact pixel at which one pin should release and the next should engage is computed twice, by two different pieces of code, at two different, loosely-sequenced moments (`OpeningScene`'s `refresh()` inside its intro `.call()`, then `StackedServicesDeck`'s later `refresh()` inside a `requestAnimationFrame`). Any bug that depends on those two computations agreeing exactly is inherently fragile under this structure.

**2. Why has this white-space issue survived multiple implementation attempts?**
Because every attempt so far has almost certainly been made *inside* one component or the other — tweaking `end` percentages, easing, or pin-spacing on the OpeningScene side, or on the StackedServicesDeck side — without touching the actual coupling point between them (the attribute/MutationObserver handshake and the two independently-timed `refresh()` calls). Local tuning inside either component can shift *where* the gap shows up or *how large* it is, but can't remove it, because the race condition lives in the seam between the two components, not inside either one's internal timeline math. A fix that doesn't touch the seam is, structurally, a fix that can't work.

**3. If you were starting from scratch today, would you build these transitions the same way?**
No. A single-viewport hero followed by a pinned multi-card deck is exactly the kind of sequence GSAP is good at when it's modeled as one continuous scroll-driven master timeline (or at minimum one ScrollTrigger with clearly labeled segments) owned by one component (or a small dedicated "HomeScrollSequence" wrapper), with the individual visual pieces (video, tagline, cards, heading) as dumb, prop/ref-driven children. The current implementation instead models it as two components each independently reinventing "when is it my turn," which is the harder problem to get right and the one most likely to regress every time either component's internals change.

**4. Is the current architecture worth preserving, or should part of it be redesigned?**
The internal work inside each component is largely fine — the video autoplay/mute-fallback state machine in `OpeningScene`, the card-stacking timeline in `StackedServicesDeck`, and the reduced-motion fallbacks in both are all reasonable, self-contained logic. What should be redesigned is the seam: the ownership of pinning/refresh/handoff between the hero and the deck. That's a boundary problem, not a problem with either component's internal animation code.

**5. What is the cleanest long-term solution?**
Introduce a single scroll-sequence owner for the homepage — a parent component (or hook) that creates one `ScrollTrigger` (or a small, explicitly ordered set of triggers it owns and refreshes together) spanning the hero-through-deck region, and that hands each section a `start`/`end` window it plays into via `.to()`/labels on a shared timeline, rather than each section instantiating its own `ScrollTrigger` and discovering the other's readiness through the DOM. Concretely, that means: one component decides pin geometry for both regions; one `refresh()` call site (or a small number, but centrally triggered, not two components independently calling it at two different moments); the intro-complete signal becomes a value passed down (or a ref/callback), not a `MutationObserver` on an attribute; and the direction-dependent side effects (muting, one-time setup guards) get replaced with logic that's an explicit, reversible function of scroll progress, so reverse scroll is mechanically guaranteed to retrace forward scroll rather than merely tending to look similar in practice.
